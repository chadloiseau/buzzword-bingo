3. **Test authentication flow** before production deployment
4. **Verify security rules** in Firebase Console Rules Playground

**SLA:** MUST fix before production deployment

### VULN-003: No Authentication Enforcement
**Severity:** CRITICAL
**Location:** [index.html:896-910](../index.html#L896-L910)
**OWASP:** A07 (Authentication Failures)

**Issue:**
Anonymous authentication is attempted but fails silently:

```javascript
auth.signInAnonymously().catch(err => {
  console.error('Anonymous auth failed:', err);
  document.getElementById('landingError').textContent = 'Connection failed. Please refresh.';
});
```

Because anonymous auth is not enabled in Firebase Console:
- Sign-in fails every time
- App continues to work (due to relaxed rules)
- No user identity established
- `playerId` uses Firebase generated UID, but it's not tied to authenticated session

**Impact:**
- Users can impersonate others
- No session management
- No way to ban abusive users
- Potential for session hijacking via URL parameter manipulation

**Remediation:**
1. Enable Anonymous Authentication in Firebase Console
2. Block UI interactions until `authReady === true`
3. Add timeout for auth failures and force page reload
4. Consider implementing proper user accounts for persistent identity

**SLA:** Fix before production deployment

## 🟠 HIGH Severity Issues (FIX BEFORE RELEASE)

### VULN-004: Session ID Predictability
**Severity:** HIGH
**Location:** [index.html:1000-1005](../index.html#L1000-L1005)
**OWASP:** A04 (Insecure Design)

**Issue:**
Session codes are generated using client-side `Math.random()`:
```javascript
function generateSessionCode() {
  const chars = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789';
  let code = '';
  for (let i = 0; i < 5; i++) code += chars[Math.floor(Math.random() * chars.length)];
  return code;
}
```

`Math.random()` is not cryptographically secure and can be predicted.

**Impact:**
- 5-character code from 33-char alphabet = ~39 million combinations
- Attackers could enumerate session codes
- With public read access (VULN-002), attackers can discover active sessions
- Session hijacking possible

**Remediation:**
1. Use `crypto.getRandomValues()` for cryptographically secure random:
```javascript
function generateSessionCode() {
  const chars = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789';
  const array = new Uint32Array(5);
  crypto.getRandomValues(array);
  let code = '';
  for (let i = 0; i < 5; i++) {
    code += chars[array[i] % chars.length];
  }
  return code;
}
```
2. Increase session code length to 8+ characters
3. Implement rate limiting on session joins
4. Add session passwords for sensitive games

**SLA:** Fix within this sprint

### VULN-005: Cross-Site Scripting (XSS) via Player Names
**Severity:** HIGH
**Location:** [index.html:1158](../index.html#L1158), [index.html:1524](../index.html#L1524)
**OWASP:** A03 (Injection)

**Issue:**
Player names are inserted into DOM using `.textContent`, which is safe. However, there's a profanity filter but **no HTML/script injection protection** if code changes to use `.innerHTML`:

```javascript
li.textContent = pdata.name;  // Currently safe
document.getElementById('winnerName').textContent = name + ' got BINGO!';  // Safe
```

**Potential Risk:**
If developers modify code to use `.innerHTML` instead of `.textContent`, XSS becomes possible.

**Current Controls:**
- Name validation: 2-20 characters
- Profanity filter
- Using `.textContent` (safe)

**Impact:**
- Currently: LOW (proper use of `.textContent`)
- If changed to `.innerHTML`: CRITICAL XSS vulnerability

**Remediation:**
1. ✅ Continue using `.textContent` for all user input
2. Add code comments warning against using `.innerHTML`
3. Implement Content Security Policy (CSP) headers
4. Add input sanitization library if HTML formatting is ever needed

**SLA:** Add CSP headers before production

### VULN-007: No Input Length Validation on Session Code
**Severity:** MEDIUM
**Location:** [index.html:1089](../index.html#L1089)
**OWASP:** A03 (Injection)

**Issue:**
Session code input has `maxlength="6"` in HTML but no server-side validation:
```javascript
const code = document.getElementById('joinCode').value.trim().toUpperCase();
if (!code) {
  document.getElementById('landingError').textContent = 'Please enter a session code';
```

**Impact:**
- HTML `maxlength` can be bypassed via browser DevTools
- Overly long session codes could cause issues
- Minor injection risk if code is used in database paths improperly

```javascript
if (!code || code.length !== 5) {
  document.getElementById('landingError').textContent = 'Session code must be 5 characters';
  return;
```

### Task 4.3: Implement Session Expiration
**Time Estimate:** 2 hours
**Owner:** Data Architect / Engineer

**Add to Firebase Security Rules:**
```json
{
  "rules": {
    "bingo-sessions": {
      "$sessionId": {
        ".read": "auth != null && (data.child('createdAt').val() > (now - 86400000))",
        // 86400000ms = 24 hours

        "createdAt": {
          ".validate": "newData.val() <= now"
        }
      }
    }
  }
}
```

**Add cleanup Cloud Function (optional):**
```javascript
// Delete sessions older than 24 hours
exports.cleanupOldSessions = functions.pubsub
  .schedule('every 24 hours')
  .onRun(async () => {
    const cutoff = Date.now() - 86400000;
    const snapshot = await admin.database()
      .ref('bingo-sessions')
      .orderByChild('createdAt')
      .endAt(cutoff)
      .once('value');

    const updates = {};
    snapshot.forEach(child => {
      updates[child.key] = null;
    });

    return admin.database().ref('bingo-sessions').update(updates);
  });
```
